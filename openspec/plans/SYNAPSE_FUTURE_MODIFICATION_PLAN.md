# Synapse 后续修改计划

## 1. 文档目的

本文用于整理 Synapse 当前版本之后的主要演进方向，明确未来修改的优先级、目标和落地顺序。

当前系统已经具备：
- 文档上传与解析
- LlamaParse 结构化解析接入
- HippoRAG 风格的 fact-first + PPR 检索链
- 候选审核/发布闭环
- Neo4j 本地 Docker 持久化骨架
- 最小图谱可视化界面
- 节点问答、节点摘要、节点笔记
- 从零开始的最小手工节点/关系/笔记入口

后续工作的重点，不再是单纯扩展底层技术栈，而是把现有能力收束成真正可用的“图式学习笔记系统”。

---

## 2. 当前产品判断

### 2.1 当前系统的优点

- 已有完整的自动入图链路：文档 -> 解析 -> chunk -> 抽取 -> 候选 -> 发布 -> 图谱
- 已有图谱增强问答能力
- 已有本地图谱可视化和节点中心交互
- 已有 Neo4j + igraph 双轨的雏形

### 2.2 当前系统的主要问题

- “从零开始记笔记”的交互体验还不够自然
- AI 回答时还没有很好地指出“与已有笔记的联系”
- 递归学习感不足，系统还更像图谱增强问答，而不是学习代理
- 图谱质量仍需继续收敛，尤其是重复节点、低价值节点和关系精度
- 节点类型、节点摘要、节点笔记尚未形成完整协同

---

## 3. 后续修改的核心目标

未来修改围绕以下四个目标展开：

1. 让系统更像“笔记工具”，而不只是“图谱系统”
2. 让 AI 回答时始终能对接已有笔记和现有图谱
3. 让图谱真正服务于学习，而不是只做展示和检索
4. 在保持当前能力可用的前提下，逐步增强递归式学习体验

---

## 4. 修改优先级

## 4.1 第一优先级：笔记本体增强

### 目标
让用户在不上传完整文档的情况下，也能把系统当作图式笔记本使用。

### 主要方向

#### A. 强化“从零开始”入口
- 支持更顺手的手工创建节点与关系
- 支持 chunk 作为原文摘录/自我理解/摘录卡片进入系统
- 支持在节点周围快速添加相关 chunk
- 节点、chunk、笔记三者形成统一入口

#### B. 强化笔记对象之间的关系
- 节点和节点之间有关系
- chunk 和节点之间有关系
- note 和节点之间有关系
- AI 在回答时要指出最相关的 node / chunk / note

#### C. 让回答与笔记显式关联
用户提问后，系统不只返回答案，还应返回：
- 最相关节点
- 最相关 chunk
- 最相关笔记
- 推荐补充的节点/关系

### 预期效果
用户会感觉自己不是在“问一个图谱系统”，而是在“围绕自己的笔记进行学习”。

---

## 4.2 第二优先级：递归学习能力

### 目标
让系统真正具备“递归式学习”的感觉。

### 当前缺口
现在的系统更像：
- 一个问题 -> 一次图检索 -> 一次回答

但递归学习应更像：
- 一个问题 -> 识别前置概念 -> 生成子问题 -> 逐层补充 -> 回答主问题

### 主要方向

#### A. 子问题机制
- 对用户问题进行问题类型判断
- 对“为什么 / 如何 / 先学什么”类问题生成子问题
- 子问题围绕局部图谱逐层展开

#### B. 图驱动递归
不是单纯生成子问题，而是：
- 根据当前节点及局部图谱
- 找到前置概念、因果链、组成链
- 让图结构本身成为递归依据

#### C. 学习状态参与递归
结合：
- 节点笔记
- 用户历史问题
- 未来的用户记忆
来判断：
- 哪些概念已掌握
- 哪些概念��要先解释
- 是否跳过某一步

### 预期效果
系统会从“给答案”进化为“带你理解”。

---

## 4.3 第三优先级：图谱质量提升

### 目标
让图真正像“学习图”，而不是“抽取结果堆”。

### 主要方向

#### A. 重复节点收敛
- 继续改进实体归一化
- 统一别名、缩写、单复数和近义表达
- 必要时支持人工 merge

#### B. 低价值节点压制
- 优先通过提示词约束抽取
- 让非知识点、数值片段、配置碎片少进入图谱
- 减少图谱噪声

#### C. 节点类型体系真正用起来
当前虽然已有 `node_type` 字段，但还未充分驱动系统行为。
未来应让不同类型节点影响：
- 前端显示方式
- 优先展示规则
- 摘要生成方式
- 学习路径排序
- 问答策略

#### D. 关系精度提升
- 关系词进一步标准化
- 弱关系压制
- 因果/依赖/组成关系优先

### 预期效果
图谱从“可用”变成“可信、可学、可组织”。

---

## 4.4 第四优先级：摘要与笔记协同

### 目标
让每个节点不只是一个点，而是一个知识卡片。

### 主要方向

#### A. 节点摘要优化
- 当前是按需 LLM 生成摘要
- 后续需要提高摘要质量
- 区分不同类型节点的摘要风格

#### B. 节点笔记增强
- 一个节点支持多条笔记已完成
- 后续应支持：
  - 笔记排序
  - 高亮重点笔记
  - 笔记与问答联动

#### C. 笔记进入问答辅助层
- 节点摘要可以直接增强正式检索
- 节点笔记则更适合作为个体化辅助上下文
- 回答时引用“你的现有理解”而不只是“原始证据”

### 预期效果
节点会更像“学习卡片”，而不是只有名字的图节点。

---

## 4.5 第五优先级：LlamaIndex 的正式定位

### 当前判断
LlamaIndex 不是当前系统的核心基础设施，但后续依然有价值。

### 适合承担的角色
- 工具编排层
- 问题分类与路由层
- 多步任务调度层
- 子问题生成与递归控制层

### 不适合承担的角色
- 替代 HippoRAG 检索主链
- 替代 LlamaParse 解析主链
- 替代 Neo4j / igraph
- 替代候选审核/发布流

### 最适合接入的时机
等“从零开始记笔记”和“递归学习”设计收敛后，再把这些能力包装成更稳定的 tool set，让 LlamaIndex 接手高层调度。

---

## 5. 推荐实施顺序

### Phase 1：让系统重新像“笔记”
1. 强化从零开始入口
2. 让回答显式关联已有 node / chunk / note
3. 让 chunk 作为笔记单元自然进入系统

### Phase 2：增强学习过程
4. 引入子问题机制
5. 基于图结构做递归式前置概念回溯
6. 让用户看到“学习过程”，而不是只看结果

### Phase 3：压实图谱质量
7. 持续优化实体归一化
8. 持续优化低价值节点过滤
9. 用好 node_type
10. 提高关系质量

### Phase 4：接回更强 agent 层
11. 收敛工具集
12. 正式接入 LlamaIndex 编排
13. 再决定是否引入更完整的记忆层与个体化学习代理

---

## 6. 当前最建议先做的事

如果只选一件事，我建议先做：

### “回答时显式指出与当前笔记最相关的 node / chunk / note”

因为这是最能立刻把产品拉回“笔记系统”的改动。

它的好处是：
- 用户能感受到系统在利用自己的笔记
- 图结构不再只是隐藏在底层
- AI 回答开始真正服务于学习与整理

---

## 7. 一句话收尾

Synapse 后续的核心任务，不是继续堆更多图算法，而是把现有 GraphRAG 能力重新收束成“图式学习笔记体验”：

- 可从零开始
- 可自动入图
- 可围绕已有笔记提问
- 可由图结构驱动学习
- 可逐步递归理解
