## 一、项目概述

### 1.1 背景与目标

当前移动端学习工具在"知识输入-理解-结构化沉淀"链路上割裂严重。本项目旨在构建面向程序员的移动端 GraphRAG 学习产品，基于微软 GraphRAG 论文范式，提供从多入口知识捕获到图谱沉淀的完整学习闭环。

**MVP 目标用户**：程序员群体（学习分布式系统、LLM 工程栈等复杂技术框架）

### 1.2 核心价值

- **知识捕获**：多入口统一采集（分享、识屏、语音）
- **图谱沉淀**：基于 GraphRAG 论文范式的实体-关系-社区层级构建
- **局部查询**：Focus View 支撑递归探索
- **全局理解**：社区摘要 Map-Reduce 回答全局问题
- **主动唤醒**：Widget/通知驱动知识复用

## 二、GraphRAG 核心流程（基于论文）

### 2.1 索引阶段 pipeline

```
源文档 → Text Chunks → 实体/关系抽取 → 知识图谱 → 社区检测(Leiden) → 社区摘要 → 建立索引
```

| 阶段 | 论文定义 | 本项目实现 |
|------|---------|-----------|
| Text Chunking | 600token chunks, 100token overlap | 文档分块与重叠设计 |
| 实体/关系抽取 | LLM 提取节点+边+claim | 实体抽取prompt + 关系定义 |
| 图谱构建 | 节点去重聚合、边权重 | 实体归一化、关系合并 |
| 社区检测 | Leiden 算法层次化划分 | C0→C1→C2→C3 层级 |
| 社区摘要 | LLM 为每个社区生成报告 | 层级化摘要生成 |

### 2.2 查询阶段 pipeline

```
用户查询 → 局部/全局模式判断 → 检索 → Map-Reduce → 生成答案
```

**局部查询 (Local)**：聚焦当前节点 + 一阶邻居 → 适用于具体问题
**全局查询 (Global)**：社区摘要 Map-Reduce → 适用于"主题是什么"类问题

| 步骤 | 论文实现 |
|------|---------|
| Map | 各社区并行生成局部答案 + 打分(0-100) |
| Reduce | 按分数排序聚合 → 最终全局答案 |

## 三、技术架构

### 3.1 分层架构

```
┌─────────────────────────────────────────┐
│           移动端 (Edge)                 │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐  │
│  │ 知识捕获 │ │ Local   │ │ 主动提醒 │  │
│  │(OCR/ASR)│ │ Query   │ │(Widget)  │  │
│  └────┬────┘ └────┬────┘ └────┬─────┘  │
│       │          │           │         │
│       └──────────┴───────────┘         │
│              端云同步                    │
├─────────────────────────────────────────┤
│           控制面 (Go)                   │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐  │
│  │ API网关  │ │ 任务编排 │ │ 通知策略  │  │
│  │ 鉴权/限流│ │ 幂等校验 │ │ 同步治理  │  │
│  └────┬────┘ └────┬────┘ └────┬─────┘  │
├───────┴──────────┴───────────┴─────────┤
│           智能面 (Python)               │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐  │
│  │ 实体抽取 │ │ 社区检测│ │ Map-Reduce│ │
│  │ 关系抽取 │ │(Leiden) │ │ 查询引擎  │  │
│  │          │ │ 社区摘要│ │ 检索重排  │  │
│  └─────────┘ └─────────┘ └──────────┘  │
└─────────────────────────────────────────┘
            ↓
     ┌──────────────┐
     │  图数据库    │
     │(Neo4j/TuGraph)│
     └──────────────┘
```

### 3.2 技术栈选型

| 层级 | 技术选型 | 职责 |
|------|----------|------|
| 移动端 | Flutter / 原生 | 知识采集、Local Query 展示、Widget |
| 控制面 | Go | API 网关、鉴权、任务编排、通知策略 |
| 智能面 | Python | 实体关系抽取、社区检测、社区摘要、Map-Reduce |
| 图数据库 | Neo4j / TuGraph | 知识图谱存储、节点/关系/社区存储 |
| 社区检测 | graspologic (Leiden) | 层次化社区划分 |
| 模型 | LLM API (OpenAI/Azure) | 实体抽取、社区摘要、答案生成 |

### 3.3 跨语言契约

- **协议标准**：统一请求体/响应体 Schema、幂等键、追踪 ID
- **错误治理**：统一错误码映射表、超时控制、熔断策略
- **版本管理**：Contract 版本号与兼容窗口规则

## 四、核心功能模块

### 4.1 知识采集 (mobile-knowledge-capture)

| 功能 | 描述 |
|------|------|
| 系统分享入口 | 接入系统分享面板，接收文本/链接/文档 |
| 语音随记入口 | ASR 转写 + 轻量实体抽取 |
| 识屏入口 | OCR 识别 + 关键实体提取 |
| 归一化服务 | 异构输入统一转换为 Text Chunk |

### 4.2 GraphRAG 索引构建 (graph-indexing-pipeline)

| 功能 | 描述 |
|------|------|
| Text Chunking | 文档分块（600token, 100 overlap） |
| 实体/关系抽取 | LLM 提取实体 + 关系 + 描述 |
| 实体归一化 | 多源实体合并去重 |
| 图谱构建 | 节点聚合、边权重计算 |
| 社区检测 | Leiden 层次化划分 (C0→C1→C2→C3) |
| 社区摘要生成 | LLM 为每级社区生成摘要报告 |

### 4.3 局部查询 (local-query)

| 功能 | 描述 |
|------|------|
| Focus View | 只展示当前节点 + 一阶邻居（Local Graph） |
| 卡片下钻 | 点击节点平滑迁移焦点 |
| 历史回退 | 保留上一焦点上下文 |
| 按需加载 | 弱网环境下可交互 |

### 4.4 全局查询 (global-query)

| 功能 | 描述 |
|------|------|
| 模式判断 | 区分 Local / Global 查询类型 |
| Map 阶段 | 各社区并行生成局部答案 + 打分 |
| Reduce 阶段 | 排序聚合生成全局答案 |
| 社区层级选择 | C1/C2 级摘要效果最佳 |

### 4.5 端云协同 (hybrid-edge-cloud-graphrag)

| 端侧 | 云侧 |
|------|------|
| OCR/ASR 转写 | 跨文档实体/关系抽取 |
| 轻量 NER 抽取 | 社区检测 (Leiden) |
| 本地 Text Chunk 缓存 | 社区摘要生成 |
| Local Query 响应 | Global Query (Map-Reduce) |
| 隐私脱敏处理 | 图谱查询与检索 |

### 4.6 主动提醒 (proactive-learning-nudges)

| 功能 | 描述 |
|------|------|
| 未探索社区识别 | 基于社区层级识别知识空白 |
| Widget 提醒 | 一跳 Deep Link 到焦点视图 |
| 关联推送 | 新旧知识高相关提醒 |
| 用户配置 | 提醒强度与静默类别 |

## 五、社区层级设计

### 5.1 层级定义

| 层级 | 含义 | 数量 | 粒度 |
|------|------|------|------|
| C0 | 根级社区 | 最少 | 最宏观 |
| C1 | 高级社区 | 较少 | 宏观主题 |
| C2 | 中级社区 | 中等 | 子主题 |
| C3 | 叶子社区 | 最多 | 细粒度知识点 |

### 5.2 社区摘要策略

**叶子社区 (C3)**：
- 聚合社区内所有节点/边/关系描述
- 优先级排序（按节点度数）依次放入 context
- 生成摘要

**高层级社区 (C2→C1→C0)**：
- 如果子社区摘要可放入 context → 直接用子社区摘要
- 否则用元素摘要 → 迭代压缩直到放入 context
- 生成更高层次摘要

### 5.3 查询层级选择

| 查询类型 | 推荐层级 | 原因 |
|----------|----------|------|
| 全局主题理解 | C1/C2 | 平衡细节与范围 |
| 细粒度探索 | C3 | 最细粒度知识点 |
| 综合回答 | Map-Reduce 多级 | 聚合多社区答案 |

## 六、查询模式设计

### 6.1 局部查询流程

```
用户查询
    ↓
判断：是否是具体实体/关系查询？
    ↓ Yes
Local检索：当前节点 → 一阶邻居 → 构建子图
    ↓
生成答案（基于子图上下文）
```

### 6.2 全局查询流程

```
用户查询
    ↓
判断：是否是全局/主题理解类查询？
    ↓ Yes
确定社区层级（C1/C2）
    ↓
Map：各社区并行生成局部答案 + 打分(0-100)
    ↓
过滤：移除打分=0的答案
    ↓
Reduce：按分数排序，依次放入context直到满
    ↓
生成全局答案
```

## 九、关键决策

1. **社区层级**：MVP 先实现 C3 叶子级 + C1/C2 高级社区
2. **查询模式**：区分 Local Query（具体问题）和 Global Query（主题理解）
3. **Map-Reduce**：MVP 采用固定层级，Beta 实现自适应选择
4. **模型策略**：MVP 优先优化抽取与摘要质量，不做训练/微调

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 社区划分质量差 | 全局答案不准确 | 调整 Leiden 参数 + 人工审核 |
| Map-Reduce 成本高 | 查询延迟高 | 层级选择优化 + 缓存 |
| 端云职责边界漂移 | 稳定性下降 | 明确接口契约与 SLA |
| 抽取精度不足 | 入图质量差 | 置信度阈值 + 人工确认 |
