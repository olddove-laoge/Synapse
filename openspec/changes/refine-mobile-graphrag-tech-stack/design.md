## Context

当前移动端 GraphRAG 方案已经完成产品能力定义，但工程实现仍面临三个实际问题：
- 语言选型未固化，导致后端与模型链路职责存在重叠风险。
- MVP 阶段若直接进入训练和微调，会显著拉长交付周期并增加不确定性。
- 跨语言服务在协议、错误处理、降级机制上缺少统一约束，可能出现联调效率低和故障扩散问题。

该设计面向竞赛版 MVP，目标是在有限周期内实现“稳定可演示、效果可解释、架构可扩展”的技术基线。

## Goals / Non-Goals

**Goals:**
- 固化 Go 与 Python 的职责分层，形成可并行开发的双服务架构。
- 建立跨语言契约规范，保证调用一致性、可观测性和故障隔离能力。
- 明确 MVP 模型策略：优先优化抽取与检索效果，不把训练作为第一阶段前置。
- 形成可演进路线，支持后续从 MVP 过渡到生产化能力。

**Non-Goals:**
- 不在本阶段定义全量微服务拆分或企业级多租户治理。
- 不在本阶段引入复杂训练平台、自动化超参搜索或大规模标注体系。
- 不在本阶段替换现有产品能力范围，仅聚焦技术栈和工程治理细化。

## Decisions

1. 架构分层：Go 控制面 + Python 智能面
- 选择：Go 承担 API 网关、鉴权、任务编排、通知策略、同步治理；Python 承担实体关系抽取、GraphRAG 推理、重排与摘要。
- 原因：Go 更适合高并发与稳定系统能力，Python 拥有最成熟的 LLM/GraphRAG 工具生态。
- 备选：全 Go 实现智能链路；否决原因是模型生态与实验效率不足。
- 备选：全 Python 统一后端；否决原因是系统治理与高并发稳定性成本更高。

2. 协议治理：统一跨语言 Contract
- 选择：定义统一请求体、响应体、错误码、幂等键和追踪 ID 标准。
- 原因：降低联调复杂度，支持端到端问题追踪和稳定降级。
- 备选：各服务自定义协议；否决原因是长期维护成本高且问题定位困难。

3. 模型策略：MVP 不以训练为前置
- 选择：MVP 先做抽取质量、检索召回与重排、上下文拼接优化。
- 原因：在竞赛周期内，这些工作投入产出比更高、可验证速度更快。
- 备选：先做 LoRA 微调；否决原因是数据准备、评测与迭代闭环成本过高。

4. 演进策略：双服务先跑通，再逐步收敛
- 选择：先完成双服务可运行闭环，再按瓶颈优化性能与端侧推理占比。
- 原因：先确保可交付，再做性能和成本精细化。
- 备选：一开始就端云全量优化；否决原因是范围过大且难以控制风险。

## Risks / Trade-offs

- [跨语言边界频繁变更] 造成开发返工 -> 通过版本化 Contract 与兼容窗口管理降低变更冲击。
- [Python 推理延迟波动] 影响交互体验 -> 增加缓存、批处理策略和超时降级路径。
- [Go 与 Python 团队理解不一致] 导致接口偏差 -> 使用共享接口文档与联调样例集进行前置对齐。
- [过早做训练导致进度失控] 影响 MVP 交付 -> 明确训练为第二阶段能力，并设定触发门槛。

## Migration Plan

1. Phase 1（MVP）
- 固化双服务边界与协议。
- 跑通输入到问答到图谱增量写入的闭环。
- 建立基础可观测性（延迟、失败率、重试率）。

2. Phase 2（稳定化）
- 针对瓶颈做重排优化、缓存策略优化、任务调度优化。
- 引入更完整的错误分类与降级策略。

3. Phase 3（增强）
- 在指标达标后评估微调与蒸馏是否带来显著收益。
- 评估端侧增强推理比例，降低云侧成本和时延。

回滚策略：
- 若 Python 智能面出现故障，Go 控制面切换到基础规则抽取与历史检索降级模式。
- 若 Contract 升级导致兼容问题，回退到上一个稳定版本并冻结新增字段。

## Open Questions

- 哪些任务可以前置到端侧执行以降低云端调用成本，同时不明显损失质量？
- 何时触发“进入微调阶段”的决策门槛最合理（例如 NDCG、人工评测分）？
